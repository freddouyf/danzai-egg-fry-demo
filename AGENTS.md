# 旦仔项目协作规范

本项目默认围绕用户的手机试玩反馈做小步迭代，目标是把“旦仔 IP”打磨成轻松、可爱、短局、适合竖屏手机的休闲游戏。

## 反馈处理

用户试玩反馈优先按四项整理：

- 最卡点：不好玩、难懂、误触、无聊的位置。
- 最爽点：已经有感觉、应该保留或放大的体验。
- 想保留：后续不要改坏的体验。
- 想删除/弱化：多余文字、机制、UI、步骤、反馈。

收到反馈后，先输出：

- 体验问题判断
- 下一轮优化目标
- 本轮不做什么
- 预计改动范围
- 是否建议 push，以及原因

## Git 策略

- 当前实验方向默认在 `prototype/rhythm-cooking` 分支迭代。
- 不要直接改 `main`，除非用户明确要求。
- 小改动默认：本地修改、`npm test`、`npm run build`、本地 commit，不 push。
- 需要手机试玩、分享给别人、阶段性存档时，再询问并执行 push。
- push 前必须明确说明本轮是否建议 push 和原因。

## 迭代节奏

默认每轮只做一个主题，例如：

- TAP / HOLD / MASH / SWIPE 手感
- B 区做菜画面
- 地图、结算、解锁
- 关卡目标和时长
- 单个新菜品或动作组合

涉及流派、角色、经济、商城等长期系统时，先做方案讨论，确认后再实现。

## 质量标准

每轮完成至少确认：

- 原读条煎蛋模式不受影响
- 节奏厨房入口正常
- 当前改动主题能实际体验到变化
- `npm test` 通过
- `npm run build` 通过
- 本地 commit 信息清楚
- 默认不 push，除非用户确认

## 默认方向

- 当前优先级：先把“节奏厨房实验模式”打磨成可验证核心玩法，再决定是否替代原读条煎蛋模式。
- 评价标准：不是功能多，而是每 10 秒内有没有明确动作反馈、成品反馈和再来一局冲动。
